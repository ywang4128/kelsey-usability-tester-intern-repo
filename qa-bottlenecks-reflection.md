# Handling QA Bottlenecks in Sprints
## Why do QA bottlenecks happen in Agile teams?
QA bottlenecks usually happen when testing starts too late in the sprint. This can be caused by developers finishing work at the last minute, the team taking on too much, or not leaving any buffer for bugs and retesting.

## How can testers avoid testing everything at the last minute?
They can spread their work across the sprint instead of waiting until the end. 

## What is parallel testing, and how does it help keep QA on track?
parallel testing means running more than one test at the same time. This helps QA get faster feedback and saves time.

## How do developers and testers collaborate to ensure smooth testing progress?
They could review stories together, write acceptance tests early, and pair test complex feature. 

## What techniques can help QA work more efficiently without compromising quality?
QA could focus on the most important area first(risk-based testing). Automation helps too, especially for repetitive regression tests. Keeping a clean data set also saves a lot of time since QA doesn't need to reset the environment often.

## If multiple features are ready for testing at the same time, how would you prioritize them?
I would prioritize features base on how risky or important they are. Take Focus Bear for example, if both the new "morning routine" and an updated "distraction-blocking rules" feature are ready at once, I would test the distraction-blocking part first because it directly affects focus for users.

## How can early involvement of QA prevent last-minute testing rushes?
QA could find issues in requirements early and prepare test cases sooner from the start of the sprint.

## What would you suggest to a team if QA keeps blocking releases due to late testing?
If QA often blocks releases, I'd suggest starting testing earlier, adding some automation to speed up checks, and planning buffer time in each sprint. Cross- training can also help, developers can run simple tests or fix failed automation quickly.
