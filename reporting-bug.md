# Bug Advocacy – Reporting Bugs Effectively #25
## What makes a good bug report vs. a bad one?
A good report is specific, readable, and action-oriented. It clearly describes what happened, how to reproduce it, and why it affects users or the project. A bad bug report is vague, emotional, or incomplete.

## What details should be included to make a bug report clear and actionable?
It sshould include clear title, steps of reproduce, environment, expected result v.s. actual result, actual impact on the user or business, severity and priority. Also screenshots, logs, or a short screen recording make it much easier for dev team to see exactly what went wrong.

## How can a tester make a bug easier to reproduce for developers?
Tester could write a simple, step-by-step instructions a developer ca follow exactly, mention any special data or conditions, and test the same steps on another device or environment to cnfirm it's not a one-off issue.

## What is "bug severity" vs. "bug priority", and how do they impact whether a bug gets fixed?
severity: how bad the bug is technically.
priority: how soon the team should fix it, based on user impact.

## How would you report a UI glitch vs. a critical login failure differently?
UI glitch:
- emphasize that the bug is cosmetic but still confusing for users.
- include screenshots or a short clip showing the layout problem
- suggest it could be fixed in the next sprint if it's not blocking core functionality
Critical login failure:
- make the title very clear and urgent
- describe the exact steps and environments affected, and emphasize the user impact
- flag it as high severity and high priority and ask the product owner to decide whether to delay the release

## If a developer dismisses a bug as "not important," how would you advocate for fixing it?
I would explain the real-world impact, show evidence, and suggest a simple fix or workaround so the team can balance speed and quality.

## How can screenshots, logs, and screen recordings improve your bug reports?
screenshots and screen recordings help developers see exactly what the tester saw. Logs give technical details about errors, network calls, or crashes that user can't see.
